**ONCOVERYX®**

**QR Code Generation System (Developer Document)**

**Unified Provider + Customer Registration Flow**

![]()

**1. SYSTEM OVERVIEW**

The Oncoveryx® platform uses a **two-level QR system**:

**A. Provider QR Code (ONE FORM SYSTEM)**

* Single registration QR for all provider types
* Used for onboarding healthcare providers

**B. Customer QR Code (MULTI-PROVIDER LINKED SYSTEM)**

* Unique QR generated per Client ID
* Used for customer test booking form access

![]()

**2. PROVIDER TYPES SUPPORTED**

All providers use the same registration system:

| **Provider Type** | **Code** |
| ----------------------- | -------------- |
| Hospital                | HOS            |
| Clinic                  | CLI            |
| Individual Doctor       | DOC            |
| Pharmacy                | PHY            |
| Diagnostic Center       | DIA            |

![]()

**3. PART A — ONE FORM QR CODE (PROVIDER REGISTRATION)**

**3.1 Purpose**

A single QR code is used for onboarding ALL provider types into Oncoveryx® system.

![]()

**3.2 Provider Registration URL**

https://yourdomain.com/provider-register

![]()

**3.3 QR CODE CONTENT**

The QR must always point to:

https://yourdomain.com/provider-register

![]()

**3.4 RULES**

* One QR for all providers
* No provider type embedded in QR
* Provider type selected inside form
* Used only for provider onboarding

![]()

**3.5 OUTPUT**

After scanning:

* Provider fills form
* System generates Client ID
* System generates unique Customer QR

![]()

**4. PART B — CLIENT ID SYSTEM (CORE LOGIC)**

![]()

**4.1 CLIENT ID FORMAT**

CLI-[STATECODE]-[YEAR]-[TYPECODE]-[SEQUENCE]

![]()

**4.2 EXAMPLES**

CLI-TN-2026-HOS-000145

CLI-KA-2026-CLI-000892

CLI-MH-2026-DOC-000321

![]()

**4.3 TYPE CODES**

| **Provider Type** | **Code** |
| ----------------------- | -------------- |
| Hospital                | HOS            |
| Clinic                  | CLI            |
| Doctor                  | DOC            |
| Pharmacy                | PHY            |
| Diagnostic Center       | DIA            |

![]()

**4.4 GENERATION LOGIC**

**Step 1: Input**

* State
* Provider Type
* System Year

**Step 2: Fetch Last Sequence**

SELECT MAX(sequence)

FROM providers

WHERE state_code='TN'

AND year=2026

AND type_code='HOS';

![]()

**Step 3: Increment**

new_sequence = last_sequence + 1

![]()

**Step 4: Build Client ID**

CLI + STATE + YEAR + TYPE + SEQUENCE

![]()

**5. PART C — CUSTOMER QR CODE (IMPORTANT)**

![]()

**5.1 PURPOSE**

Each Client ID generates a **unique QR code for customer test booking form access**.

![]()

**5.2 CUSTOMER REGISTRATION URL**

https://yourdomain.com/register?client_id=CLI-TN-2026-HOS-000145

![]()

**5.3 QR CODE CONTENT**

Each provider gets a unique QR:

https://yourdomain.com/register?client_id=<CLIENT_ID>

Example:

https://yourdomain.com/register?client_id=CLI-TN-2026-HOS-000145

![]()

**5.4 RULE**

* Every Client ID MUST generate a unique QR
* QR is immutable once created
* QR is linked to provider only

![]()

**6. CUSTOMER REGISTRATION FORM (ONCOVERYX®)**

![]()

**6.1 SYSTEM RULES**

* Provider selection is automatic
* No manual provider selection allowed
* Must use client_id from QR

![]()

**6.2 CORE FLOW**

QR Scan → client_id → Provider Mapping → Registration Form → Booking

![]()

**6.3 VALIDATION RULE**

* If client_id invalid → BLOCK access
* Provider fetched from DB only

![]()

**10. SECURITY (VERY IMPORTANT)**

![]()

**10.1 SIGNED QR URL (RECOMMENDED)**

https://yourdomain.com/register?client_id=XXX&token=HMAC

![]()

**10.2 TOKEN GENERATION**

token = HMAC(client_id + secret_key)

![]()

**10.3 VALIDATION**

On scan:

1. Validate client_id
2. Validate token
3. Fetch provider
4. Load form

If invalid:

* Block access

![]()
