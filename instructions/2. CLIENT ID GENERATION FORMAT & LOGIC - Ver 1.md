__CLIENT ID GENERATION FORMAT & LOGIC__

__Healthcare Provider Unique Identity System__

__1\. OBJECTIVE__

To generate a __unique, structured, and traceable Client ID__ for every healthcare provider \(Hospital, Clinic, Doctor, Pharmacy, Diagnostic Center\)\.

The Client ID must:

- Be globally unique
- Be human\-readable
- Encode basic metadata \(region \+ year\)
- Work seamlessly with QR tracking
- Support analytics and referral tracking

__2\. FINAL CLIENT ID FORMAT \(RECOMMENDED\)__

__FORMAT STRUCTURE__

CLI\-\[STATECODE\]\-\[YEAR\]\-\[TYPECODE\]\-\[SEQUENCE\]

__EXAMPLE__

CLI\-TN\-2026\-HOS\-000145

CLI\-KA\-2026\-CLI\-000892

CLI\-MH\-2026\-DOC\-000321

__3\. FIELD BREAKDOWN__

__Segment__

__Meaning__

State abbreviation \(TN, KA, MH, etc\.\)

__5\. GENERATION LOGIC \(BACKEND\)__

__STEP 1: INPUT DATA__

When provider registers:

- State
- Provider Type
- Year \(system date\)

__STEP 2: FETCH LAST SEQUENCE__

Query database:

SELECT MAX\(sequence\)

FROM providers

WHERE state = 'TN' AND year = 2026 AND type = 'HOS';

__STEP 3: INCREMENT SEQUENCE__

last\_sequence \+ 1

Example:

- Last = 144
- New = 145

__STEP 4: BUILD CLIENT ID__

CLI \+ STATE \+ YEAR \+ TYPE \+ SEQUENCE

Final:

CLI\-TN\-2026\-HOS\-000145

__6\. RULES \(VERY IMPORTANT\)__

__RULE 1: GLOBAL UNIQUENESS__

No two providers can share same Client ID\.

__RULE 2: IMMUTABLE ID__

Once generated:

- Cannot be changed
- Cannot be reassigned

__RULE 3: SEQUENTIAL PER CATEGORY__

Sequence resets per:

- State
- Year
- Provider Type

Example:

- TN\-2026\-HOS\-000001 starts fresh every year

__RULE 4: ZERO GAPS \(OPTIONAL\)__

If a record is deleted:

- Do not reuse ID
- Maintain audit integrity

__8\. QR CODE INTEGRATION \(IMPORTANT\)__

Each Client ID generates a QR:

__QR CONTENT:__

https://yourdomain\.com/register?client\_id=CLI\-TN\-2026\-HOS\-000145

__9\. SECURITY RECOMMENDATION__

To prevent tampering:

- Sign Client ID using HMAC token \(recommended\)
- Example:

CLI\-TN\-2026\-HOS\-000145?token=abc123secure
