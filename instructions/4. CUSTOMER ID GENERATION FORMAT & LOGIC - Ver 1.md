__CUSTOMER ID GENERATION FORMAT & LOGIC__

__Patient / Customer Unique Identity System__

__1\. OBJECTIVE__

To generate a __unique, traceable Customer ID__ for every patient/customer booking\.

The Customer ID must:

- Be unique across the system
- Be linked to a Provider Client ID
- Support booking, sample collection, and report tracking
- Be simple enough for staff to use
- Be scalable across cities and years

__2\. FINAL CUSTOMER ID FORMAT \(RECOMMENDED\)__

__FORMAT STRUCTURE__

CUST\-\[CLIENTSHORT\]\-\[YEAR\]\-\[TYPE\]\-\[SEQUENCE\]

__EXAMPLE__

CUST\-ABC\-2026\-BLD\-000251

CUST\-TN01\-2026\-BLD\-001102

CUST\-HSP5\-2026\-BLD\-000045

__5\. CLIENTSHORT GENERATION RULE__

Derived from Provider Client ID:

Example:

CLI\-TN\-2026\-HOS\-000145

Convert to:

HOS145

Rules:

- Remove prefixes \(CLI, state, year\)
- Keep provider type \+ last digits
- Keep it short for readability

__6\. GENERATION LOGIC \(BACKEND\)__

__STEP 1: INPUT DATA__

When customer registers:

- Client ID \(from QR\)
- Service type \(Blood/Home/Clinic\)
- Current year

__STEP 2: FETCH CLIENT SHORT CODE__

From provider table:

CLI\-TN\-2026\-HOS\-000145 → HOS145

__STEP 3: FETCH LAST SEQUENCE__

Query:

SELECT MAX\(sequence\)

FROM customers

WHERE client\_short = 'HOS145'

AND year = 2026;

__STEP 4: INCREMENT SEQUENCE__

last\_sequence \+ 1

Example:

- Last = 250
- New = 251

__STEP 5: GENERATE CUSTOMER ID__

CUST \+ CLIENTSHORT \+ YEAR \+ TYPE \+ SEQUENCE

Final:

CUST\-HOS145\-2026\-BLD\-000251

__7\. RULES \(IMPORTANT\)__

__RULE 1: GLOBAL UNIQUENESS__

No duplicate Customer IDs allowed\.

__RULE 2: IMMUTABLE ID__

Once generated:

- Cannot be edited
- Cannot be reassigned

__RULE 3: CLIENT BOUNDARY__

Sequence is maintained per:

- Client Short Code
- Year
- Service Type

__RULE 4: MULTIPLE BOOKINGS__

Same patient can have multiple IDs if:

- Different visits OR
- Different service types \(optional rule choice\)

Recommended:

- Keep one customer ID per booking \(safer for diagnostics\)
